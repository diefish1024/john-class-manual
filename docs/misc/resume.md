## 一、背景

### Resume 还是 CV？

在英文语境下，**Resume** 与 **CV (Curriculum Vitae)** 是两种不同的文档：

- **Resume**：业界求职用，**严格 1 页**，强调"对方在意什么"，按目标岗位裁剪内容。
- **CV**：学术求职 / 申请用，**1–N 页**（PhD 申请通常 1–2 页，正教授可能 10+ 页），按时间或类别完整罗列教育、研究、出版、Talk、Award、Service 等。

中文社区习惯把两者都叫"简历"，本章节也沿用这一称呼，但会在具体建议中区分。除非特殊说明，"简历 (1 页)"默认指 Resume，"CV"默认指学术 CV。

**中文简历** 与 **英文简历** 也存在差异：

- 中文简历常见于国内大厂实习投递、国内直博 / 保研夏令营、部分国企央企岗位。
- 英文简历 / CV 用于海外申请、外企、开源社区与远程岗位。
- 同一段经历的中英文版本，建议**分开维护两份源文件**，而不是逐字翻译——措辞、详略程度、模块顺序在两种语言里都不一样。

### 简历不一定是对口的人看的

一个常被忽略的信息差是：**简历最先看到的人，往往不是对口技术面试官**。常见的初筛者依次是：

- HR / 半技术向 HR
- 机筛系统（关键词匹配）
- **LLM 筛简历**（已有大厂在使用）

即使过了初筛进入面试，**很多面试官也是面试当场才第一次看你的简历**。一份不专业、技术点表述模糊的简历，会让面试官和你"鸡同鸭讲"，你有力使不出。

所以一份简历需要同时满足两个底层目标：

1. 让 **不了解技术的人（HR / 机器）** 觉得你专业、关键词匹配。
2. 让 **当场扫简历的面试官** 一眼能抓到可问的点（参见后文 [Hook 写法](#3-hook-写法)）。

### 典型应用场景

| 场景 | 文档类型 | 重点 |
|------|---------|------|
| 北美 / 欧洲 / 港新 PhD 申请 | 英文 CV (1–2 页) | research fit、出版物、推荐信线索 |
| 暑研 / RA 申请 | 英文 CV (1–2 页) | 与目标实验室方向的契合度 |
| 国内大厂 / 创业公司实习与全职 | 中 / 英文 Resume (1 页) | 工程能力、项目可量化结果 |
| 量化 / 金融岗 | 英文 Resume (1 页) | 数学物理基础、竞赛奖项 |
| 国内直博 / 保研夏令营 | 中文简历 + 成绩单 | GPA、排名、科研经历 |

### 为什么要尽早开始

**真正的功夫在简历之外**——大三大四临阵磨枪很难写出有竞争力的简历，简历是项目、论文、实习本身质量的*结果*，不是动作。对业界岗，重要性大致是：

> **实习经历 > 项目经历 ≈ 竞赛 / 科研（特别突出可破例）> 学校**

所以一旦有过对口实习 / 科研，详细写就行，普通项目随便写——有实习的情况下 90% 的面试时间都会聊实习。

建议从大一 / 大二开始：

1. 维护一份长期更新的"经历流水账"，记录做过的项目细节、关键数据、commit 链接、报告链接，写简历时才有素材可筛选。
2. 每完成一段经历（课程项目结题、论文投出、实习结束）就**立刻更新**简历，比几个月后再回忆要靠谱得多。
3. 顺手维护一个**个人主页**（GitHub Pages / 个人域名），把简历挂上去——可以实时更新、便于教授直接查阅，也可以通过访问日志辅助判断对方是否查看了申请材料。

## 二、模板选择

技术岗简历建议直接使用 LaTeX（或 Typst）撰写：版式稳定，PDF 输出在不同设备上效果一致。不熟悉本地 LaTeX 环境的同学，可以直接用 [Overleaf](https://www.overleaf.com/) 在线编辑。

### 常见英文模板

- [Awesome-CV](https://github.com/posquit0/Awesome-CV) — GitHub 上最流行的 LaTeX 简历模板之一，提供传统单栏与现代分栏两种风格，定制性强，适合技术岗求职。
- [Jake's Resume](https://github.com/jakegut/resume) — 极简单栏模板，近两年北美求职社区出镜率很高，1 页内信息密度高。
- [ModernCV](https://github.com/moderncv/moderncv) — 老牌学术 CV 模板，适合 1–2 页 PhD 申请向 CV。
- Typst 社区中有不少轻量模板，编译快、语法相对友好，适合频繁迭代。

### 常见中文模板

- [hijiangtao/resume](https://github.com/hijiangtao/resume) — 国内求职社区出镜率很高的中文 LaTeX 简历模板。
- 大多数英文模板替换字体（思源宋体 / 思源黑体）后即可用于中文，但需要注意：
    - 用 XeLaTeX / LuaLaTeX 编译，确保 CJK 字体支持。
    - 中英文字号通常不同，建议中文比英文小 0.5–1 pt，否则视觉上中文字会偏大。
- 国内简历模板（如「五百丁」「超级简历」等）有大量在线模板，但样式良莠不齐，仍建议参考下文反例。

### 个人主页

有条件的同学建议同时维护一个个人主页放置在线版简历。常见方案：

- GitHub Pages + Jekyll / Hugo / Hexo
- [academicpages](https://github.com/academicpages/academicpages.github.io)（学术主页常用）

### 一些反例

挑模板和改样式时，建议避开：

- 通篇红黄蓝绿的多色配色，强调色超过两种就会像传单。
- 技能条形图、五角星打分、雷达图、圆形头像描边等装饰性图形，主观且没什么信息量。
- Comic Sans、楷体、行书、隶书、各类「艺术字」字体；中英文混排时建议成对挑选。
    - 英文常见的搭配有 Latin Modern Roman / Computer Modern / Charter / EB Garamond，或无衬线的 Helvetica / Source Sans。
    - 中文可以选思源宋体 / 思源黑体，避免系统默认宋体在小字号下糊成一团。
- 正文小于 10 pt、行距过窄，整页密不透风。
- 页眉页脚塞水印、班级口号、二维码等无关元素。
- **单独成一级标题的"技术栈"模块**——详见第四节[关于技术栈模块的争议](#关于技术栈--专业技能模块的争议)。
- 除非投递方明确要求，技术岗一般不需要贴照片；若贴，使用正式证件照。

文件命名规范化即可，如 `Zhang_San_Resume_2026.pdf` 或 `Zhang_San_CV_Stanford.pdf`。

## 三、好的例子

最直接有效的范例来源是**已经走通这条路的人**。建议从下面几个渠道找参考：

### 找范例的渠道

- **目标项目里已录取的师兄师姐主页**：在你想申请的实验室或部门，找最近几届录取者的个人主页，他们挂的 CV 就是最贴近"成功样本"的参考。
- **目标方向的博士生 / 青年教授 CV**：例如 [Sacha Epskamp](http://sachaepskamp.com/) 这类被反复推荐的范例。注意要看**博士生的早期 CV**，而不是已成名教授的"成熟版"CV，前者更接近你目前的状态。
- **CS-SoP 等社区**：[cs-sop.org](https://cs-sop.org/) 收录了大量申请成功者的材料。
- **一亩三分地、Reddit r/cscareerquestions** 等论坛：有大量大厂简历修改贴，按方向（SDE / MLE / Quant）分类查找。
- **水源相关帖子**：详见 [简历指北](https://shuiyuan.sjtu.edu.cn/t/topic/397988) 。

### 申请者 CV vs 成熟 CV 的区别

注意：**申请者 CV 与教授 / 资深博士生 CV 的写法不同**。

- 教授 CV 只需列出论文标题、会议、年份即可——读者默认知道这些工作的分量。
- **申请者 CV 在每段经历下要补充"做了什么"的细节**，用 bullet 突出技能与贡献。
- 申请者还可以放上**未发表但能体现能力的课程项目**，这是教授 CV 不会出现的内容。

### 示例方向（待补充）

!!! tip "待补充"
    欢迎学长学姐提供匿名化处理过的简历范例，附上简短的「适用场景 / 当时投递结果」说明。

- 示例方向 1：**北美 PhD 申请向 CV** — 以教育背景、科研经历、出版物、Talk、Award 为主，长度 1–2 页可接受。
- 示例方向 2：**国内大厂实习简历** — 以项目 / 实习 / 技能栈为核心，强烈建议 1 页。
- 示例方向 3：**量化 / 金融岗简历** — 突出数学物理基础、竞赛奖项、相关项目；版式极简。
- 示例方向 4：**暑研 / RA 申请简历** — 介于学术 CV 与实习简历之间，强调与目标实验室方向的契合度。
- 示例方向 5：**国内保研 / 直博中文简历** — 重点突出 GPA、排名、科研经历、课程项目。

## 四、简历格式

### 长度与篇幅

- **业界 Resume（实习 / 全职）**：严格 **1 页**。超过 1 页几乎一定意味着内容没有筛选。
- **学术 CV（PhD 申请 / 暑研）**：可 **1–2 页**；出版物 / Talk 较多可适当延伸，但仍要避免无意义的填充。
- **中文保研简历**：通常 1–2 页，因需要列出课程成绩 / 排名详情，比英文 Resume 略宽松。

### 字号与排版的具体推荐

- 正文 **10–11 pt**，姓名标题 **18–24 pt**。
- 行距 **1.0–1.15**。
- 页边距 **0.5–0.75 in**（约 1.3–2 cm）。
- 中英文混排时，中文字号比英文小 0.5–1 pt，视觉上更平衡。

### 常见模块顺序

以下顺序仅供参考，**真实顺序应当根据"你想让对方先看到什么"来排**（参见下文 [价值排序](#七价值排序)）：

1. **个人信息（Header）**：姓名、邮箱、电话（可选）、个人主页 / GitHub / LinkedIn。
2. **教育背景（Education）**：学校、学院、专业、入学 / 毕业时间、GPA / Rank（如果亮眼）。
3. **科研经历（Research Experience）** *或* **实习经历（Work / Internship Experience）**。
4. **项目经历（Projects）**。
5. **出版物（Publications）** / **奖项（Awards）** / **竞赛（Competitions）**。
6. **技能（Skills）** / **语言（Languages）**。

学术 CV 还可能包含：**Talks / Presentations**、**Teaching Experience**、**Service**（如审稿、组织 seminar）、**References**（推荐人联系方式，部分场景需要）。

### 关于"技术栈 / 专业技能"模块的争议

不少同学习惯用一整块技术栈（C++ / Rust / Go / Python / Docker / K8s / Kafka / ...）填满第一页，认为可以**对抗机筛**。但实践中：

- 面试官提问的逻辑是 **"从你简历上自己写的点开始广度优先进攻"**。技术栈写得越多，等于告诉面试官"我有 N 个弱点快朝这打"。
- 真正强的候选人正文里几条经历就够分量了，根本不需要报菜名。
- 一种更成熟的做法：**直接废除独立的技术栈模块，让面试官顺着你的主推经历深入**，而不是被报菜名一黑接一黑地否定。

折中策略：业界岗为了防机筛，可以保留一个**精简且全部能扛住追问**的技术栈（4–8 个词），并把它放在不那么显眼的位置；学术 CV 一般不需要这个模块。

### 排版细节

- 每一段经历采用「**机构 / 职位 — 时间**」对齐，时间靠右。
- 同一份简历中所有日期格式统一（如 `2024.09 – Present` 或 `Sep 2024 – Present`，**不要混用**）。
- 项目符号统一（`•` 或 `–`，不要既有圆点又有方块）。
- 列表条目尽量保持 **每条 1 行，至多 2 行**，避免出现"孤儿单词 / 寡妇行"（英文术语 *widow / orphan*，可搜 LaTeX 解决方案，如 `\widowpenalty`、`microtype` 宏包）。
- 中英文混排时注意空格规则（中文与英文 / 数字之间留半角空格）。
- **超链接**：邮箱、个人主页、GitHub、论文 `[pdf]/[code]/[slides]` 都建议加上可点击的超链接（LaTeX 中用 `hyperref`），方便对方一键访问。
- PDF 导出后在不同设备 / 系统上预览一遍，注意字体回退（fallback）问题——中文字体在国外评审的电脑上可能显示异常，建议**嵌入字体**（XeLaTeX 默认嵌入）。

## 五、两条核心原则

在进入具体写作技巧之前，先记住两条原则。后面所有的方法——STAR、Hook、抛名词、列指标——都是这两条原则的衍生物。

> **原则一**：没有人在意你*会什么*，在意的是*你能给对方带来什么*（能干什么活）。
>
> **原则二**：**没写的默认不会，写了讲不出来的默认是抄的。**

第一条决定了**写什么**：罗列技能没用，要把技能落到"我用它做出了什么可被验证的成果"上。

第二条决定了**怎么写**：简历是一份"对自己负责的清单"——写上去的每一项都默认你能在追问下扛住。讲不清楚 baseline 的数字、答不上来的技术栈、解释不了选型的项目，比不写更糟。

## 六、编写思路

写简历不是把自己做过的事情都列出来，而是要让对方在很短时间里判断出你和 TA 的需求是否匹配。

### 1. 从「需求」出发

动笔之前先想清楚：这份简历投给谁、对方在意什么、TA 大概会花多长时间看完。教授、HR、业务面试官关心的点完全不同，针对不同投递目标准备不同版本的简历是值得的。

研究表明，HR / 招生委员会平均看一份简历不超过 **30 秒**，目光呈 **F 型**扫过左上区域。因此**第一页前 1/3** 是黄金区域，必须放最能打的内容。

### 2. 突出重点，砍掉噪声

- 与目标无关的经历（如和 CS 完全无关的社团水奖）可以删掉。
- 同一类经历保留最有代表性的两三条即可，不要堆砌。
- 一段经历的 **bullet 数建议 3–5 条**，每条对应一个真实可被追问的技术点。再多就显得在偷同事文档。

### 3. 用「成果」而非「职责」描述经历（STAR 法则）

每条 bullet 建议套用 **STAR 法则**：

- **S**ituation：背景（什么数据集 / 什么系统 / 什么团队规模）
- **T**ask：你要解决的问题
- **A**ction：你具体做了什么（用强动词开头）
- **R**esult：可量化的结果

对比：

- 反例：「负责模型训练与调参」
- 正例：「在 XX 数据集（1.2M 样本）上将 baseline 准确率从 78% 提升至 85%，方法被合并入团队主分支」

### 4. 措辞

- 使用强动词，如 `Designed / Built / Optimized / Led / Analyzed / Proved`，避免 `Did / Worked on / Helped with` 之类的弱表达。
- 中文简历也建议以动词开头（"设计了…"、"实现了…"），省略主语"我"。
- 避开「具有良好的沟通能力」「热爱学习」这类不可证伪的空话，简历上几乎不会加分。
- 缩写要看场合：`RLHF`、`MoE`、`KV Cache` 等术语在学术 CV 中可以保留，业界简历中酌情展开。

### 5. 一致性

- 时态一致：英文简历中已结束的经历用过去式，进行中的用现在时。
- 主语隐去：英文 bullet 通常省略 "I"，以动词开头。
- 术语一致：同一个工具 / 框架在全文中使用相同写法（如统一为 `PyTorch`，不要混用 `pytorch` / `Pytorch`）。
- 中英文版本之间，模块名、项目名、组织名的译法要稳定（"清华大学计算机系" ↔ "Department of Computer Science, Tsinghua University"，全文统一）。

### 6. 关键词扫描

写完后用关键词搜索自己的简历，检查目标 JD / 教授主页上的核心关键词是否出现过。例如教授主页强调 *diffusion models for protein design*，你简历里若只有 *generative model* 就应该改写得更具体。

## 七、价值排序

简历最容易被忽略的一点是：把「对方最在意的、你最强的」放在最显眼的位置。所谓「显眼」，通常指**第一页上半部分**、**同一段经历内部的第一个 bullet**、整份简历中靠前的模块。

### 几条具体经验

- GPA / Rank 高就放教育背景的第一行；不亮眼就弱化甚至省略（学术申请一般无法省略）。
- 有顶会论文或知名实习就优先放最前面，哪怕打破"标准模块顺序"。
- 同一段经历的 bullet 也要排序：最能打的成果放第一条，琐碎的工程细节放最后。
- 同一份经历投不同方向时可以重新排序，例如科研申请时强调「实验设计 / 学术贡献」，业界投递时强调「工程实现 / 系统优化」。

### 业界岗 vs PhD 申请的两套排序

**业界实习 / 全职**：

> 实习经历 > 项目经历 ≈ 竞赛 / 科研（科研特别突出可破例）> 学校

**PhD 申请**：

> CV（研究经验）+ 研究相关推荐信 ≫ GPA > GRE > TOEFL

因此 CV 上**研究经历 / 出版物 / Talk** 应当尽量前置；GPA 如果不亮眼，可以放在 Education 块内部不突出处理，把版面让给科研。

### 本手 + 妙手

一份合格的简历通常由两层构成：

- **本手**：符合 JD 的基础项。投后端就该有后端项目，投算法就该有 ML 项目；缺了会被怀疑"没基础"。
- **妙手**：差异化亮点。在满足本手的前提下，挑一两个**面试官未必精通的方向**展开，例如投 Golang 中台时聊 AutoML / kubeflow / k8s 新特性，把面试节奏带到你最熟的领域。

只有本手 → 平庸；只有妙手而本手缺失 → 被怀疑跑题。两者都要。

### 自检方法

写完之后可以让一个不太了解你的朋友扫一眼简历（30 秒为限），问 TA 印象最深的是什么。如果不是你最想突出的，那就需要再调整。

## 八、修饰技巧

修饰不等于造假——但要承认一个现实：**面试官也不全知**，你完全可以、也应该用合理的写法引导面试节奏。修饰的边界是「事实正确 + 自己能扛住追问」。

### 1. 针对性（最重要）

投递前花几分钟看一下目标岗位的 JD 或教授的研究主页，把简历里相关关键词向对方的措辞靠拢。例如教授主页强调 *diffusion models for protein design*，而你做过相关项目，就明确写 *diffusion model*，而不是泛泛的 *generative model*。

- 同一段经历投不同方向时，bullet 顺序可以重排，把最契合对方需求的成果放第一条。
- 不同方向准备不同版本（PhD 申请版 / 实习版 / 量化版），可以共用源文件，用 LaTeX 的 `\if` 条件编译切换模块。

### 2. 列指标（极其重要）

**没有数字的简历，在工业岗会被默认是 fake 项目。**

- 每个项目至少要有一个**可量化的核心指标**：QPS、P50/P90/P99 延迟、准确率提升、数据规模、加速比、用户数。
- 不要写"平均响应时间 < 2s"这种业余表述，要用 **P50 / P90 / P99** 这种工业化说法。
- 指标要能说清 **baseline、测试集、硬件配置、对比对象**。
- 如果对比的是 **论文实现 / 开源库 / 上一版本**，更是大亮点。
- **关键提醒**：网上项目辅导太多，"答不出基础指标"会让面试官直接判定你整个项目是 fake 的。所以**平时做项目就要有意识地记录指标、跑 benchmark**，临投递前补不出来的。

关于「超过 baseline」：

- 面试官不一定知道你的小领域。
- **基线是你自己选的**——后端可以拿之前的版本做基线，算法可以拿你能打过的 baseline。
- 关键是说清**你的 tradeoff 和创新点**，而不是宣称 SOTA。

### 3. Hook 写法

"Hook"指**刻意把简历某些表述写得稍微模糊或抽象，让面试官顺着你设的钩子提问**，把面试引到你最熟的领域。

- 反例：「熟悉数据库技术，了解 MySQL」——面试官只能问 MySQL 八股，方向被对方控制。
- 正例：「熟悉数据库技术，了解现有 NoSQL & SQL 范式，例如 MongoDB、MySQL」——面试官大概率会问"讲讲 NoSQL"，你就可以从选型理由、业界方案、自己手写过的 KV 存储、LSM 键值分离一路展开。

Hook 的本质是**把面试官的提问引到你准备最充分的方向**，而不是被动等他随机抽八股。

**使用前提**：你必须在被钩到的话题上有真实储备，否则就是给自己挖坑。

### 4. 抛名词

对于实在没什么亮点的项目（如烂大街的电商 / 书城），可以在第一行直接列出**完整技术栈关键词**：

> `SpringBoot + React + Kafka + MongoDB + Neo4j + Redis + WebSocket + Spark`

作用是：

- **快速信息交换**：让面试官 1 秒判断你这个项目大致用了什么。
- **防机筛**：关键词命中 ATS / LLM 筛选器。
- **凑字数**：弱项目无亮点时，名词堆叠总比空着好。

同样要遵守 Hook 的前提：**写上去的每个名词都要能扛住追问**。

### 5. 包装的边界

包装可以，但守住两条线：

1. **你要真懂**——把"prompt 工程"包装成"显式 / 隐式分析和思维链引导"，行内人一眼能看出是拙劣包装；用错术语比用朴素表达更扣分。
2. **不要为了添彩偏离主线**——一个 3 个月的实习不可能同时干完 SFT、RL、算子解析、记忆工程、planner、库开发。写多了反而显得在偷同事文档。

### 6. 视觉

- 用加粗突出关键词（项目名、技术栈、奖项名），但整页加粗不要超过 **10–15%**，否则就等于没有加粗。
- 左右对齐分割信息块（左侧是「做了什么」，右侧是「时间 / 地点」），可以减少阅读跳跃。
- 适当留白，比起密不透风更让人愿意看。

### 7. 迭代与版本管理

- 每改一稿都导出 PDF 再读一遍，屏幕上看不出的排版问题在 PDF 里通常很明显。
- 请不同背景的人 review：同方向的同学看技术细节，不同方向的朋友看可读性和价值排序。
- 用 Git 管理简历源码，每次投递前打 tag（如 `v2026-google-intern`、`v2026-stanford-phd`），方便复盘哪一版投了哪里。

### 8. AI 润色的边界

LLM 适合做：

- 语法检查
- 强动词替换（`Did` → `Designed / Built / Optimized`）
- 消除中式英语 / 不自然搭配

LLM **不适合**做：

- 直接生成项目描述——它会编造细节、产生千篇一律的句式（如 *leveraging cutting-edge technologies*、*demonstrating strong analytical skills*），评审一眼就能看出"AI 味"。
- 自动量化结果——它不知道你真实的 baseline 和数据规模，编出来的数字会害死自己。

## 九、常见反例集锦

以下是从真实简历评议中归纳出的高频反例，每条对应一个具体问题：

- ❌ **"负责开发者平台的可视化组件开发"** → 太职责描述，无技术深度。改成"独立设计 / 实现 XX 组件，支持 YY 场景，月活 ZZ"。
- ❌ **"撰写 PRD 并把控了鉴权系统的设计"** → 写 PRD 不是技术点；"把控"了能讲清 OAuth / 一致性 / 多设备登出吗？
- ❌ **"了解 Go-kit / K8s / OpenTelemetry / Spring AI / LangChain / LangChain-Go..."** → 报菜名 + 重复项，立靶子。
- ❌ **"在 12 轮对话测试中准确率 90.1%"** → 什么是"12 轮对话测试"？为什么是 12？外部 benchmark 有数据吗？
- ❌ **"上下文相关性提升至 91.2%"** → 原先多少？怎么评的？向量相似度还是 LLM as judge？
- ❌ **"平均响应时间 < 2s"** → 业余表述，应该用 P50 / P90 / P99。
- ❌ **"每段经历只有一个 item，且只写了一个月"** → 不如不写，会被怀疑在集邮 / 凑数。
- ❌ **"顶会论文写得春秋笔法，还放在简历最底下"** → 含金量最高的应该放最顶上，必要时打破标准模块顺序。
- ❌ **"校园经历放在学术经历前面"** → 面试官第一眼看到"xx 社团负责人"，会质疑你来干嘛。
- ❌ **"把 GitHub 链接做成隐藏链接生怕面试官看到"** → 反向操作，绝对错误实践。GitHub 主页应该作为加分项主动暴露。
- ❌ **"`prompt 工程` 被包装成 `显式 / 隐式分析和思维链引导`"** → 用错术语，行内人一眼看穿。
- ❌ **"三个月实习写了 6 个完全独立的大模块"** → 边界模糊，被怀疑挂他人成果。

## 十、自检清单

投递前请逐项过一遍：

- [ ] 简历上的每一项技术栈，被追问时都能答上来？
- [ ] 每个数字都能说清 baseline、测试集、硬件、对比对象？
- [ ] 用的是 P50 / P90 / P99，而不是「平均响应时间」？
- [ ] 所有日期格式统一（如 `Sep 2024 – Present`，不要中英混用）？
- [ ] 中英文之间都留了半角空格？
- [ ] 项目符号统一（不要既有 `•` 又有 `–`）？
- [ ] 文件名规范（`姓名_岗位_日期.pdf`）？
- [ ] 在不同设备 / 系统上预览过 PDF（字体回退问题）？
- [ ] 邮箱专业（避免 QQ 邮箱昵称怪异；首选学校邮箱或 Gmail）？
- [ ] 所有超链接都能点击且指向正确？
- [ ] GPA / 排名如果写了，能在面试里说清算法（加权 / 算术 / 主修）？
- [ ] 每段经历的第一个 bullet 是不是这段经历里最能打的成果？
- [ ] 整页加粗比例不超过 15%？
- [ ] 没有「具有良好的沟通能力」「热爱学习」这类不可证伪的空话？
- [ ] 每段经历 bullet 数在 3–5 条，没有为了凑数堆叠？
- [ ] 中英文版本的项目名、机构名译法一致？
- [ ] 本手齐全（JD 要求的基础项都有）且至少有一个妙手？

## 十一、长期维护

简历是一份**长期维护的文档**，不是临投递前的一次性产物。

实习 / 科研期间，每天醒来应该想两件事：

1. **今天去哪里搞产出**。
2. **我的简历可以更新什么**。

> 优秀的简历都是平时维护出来的，不是临投递前堆出来的。

具体做法：

1. **每完成一段经历**（项目结题、论文投出、实习结束）就立刻更新简历主线。
2. **维护一份"经历流水账"**（Notion / Obsidian / 仓库里的 markdown）记录每段经历的细节、链接、数据，写简历时从中筛选，而不是凭回忆。
3. **同时维护多个版本**：英文 Resume / 英文 CV / 中文简历至少各一份，长期同步更新。
4. **每年自检一次**：即使没有投递计划，每年至少打开一次简历，删掉过时内容、加上新经历。
